### Tamper Simulation – Swapped Private Keys

**Date:** 2026-03-01  
**Status:** ✅ Gate 2 CONFIRMED WORKING

#### Method

Swap `client-0.private.pem` ↔ `client-1.private.pem` on disk.  
Each client then signs its weight update with the **other** client's private key, which will not verify against the server's stored public key for that client.

```powershell
# Swap keys
$tmp = Get-Content signing_keys\client-0.private.pem -Raw
Copy-Item signing_keys\client-1.private.pem signing_keys\client-0.private.pem
Set-Content signing_keys\client-1.private.pem $tmp

# Run
docker compose up data-init
docker compose up --abort-on-container-exit server client-0 client-1

# Restore
Copy-Item signing_keys\client-0.private.pem.bak signing_keys\client-0.private.pem -Force
Copy-Item signing_keys\client-1.private.pem.bak signing_keys\client-1.private.pem -Force
```

#### Key Log Lines (Rejection proof)

```
fl-server | INFO :      aggregate_fit: received 1 results and 1 failures

fl-server | WARNING :   [Gate 2] ✗ REJECTED tampered update from client_id=1
fl-server | [Gate 2] Round 1: 0 accepted, 1 rejected
fl-server | WARNING :   [Gate 2] No valid updates this round – skipping aggregation
```

#### Observations

- **client-1** signed its update with `client-0`'s private key (wrong key). The server held `client-1`'s public key and the signature did not match → `REJECTED tampered update from client_id=1`. ✓
- **client-0** signed with `client-1`'s private key (wrong key). Its gRPC connection experienced the pre-existing transient `TSI_DATA_CORRUPTED` SSL dropout (same as observed in Phase 2), which caused Flower to count its update as a transport-layer `failure` before Gate 2 could evaluate it. This is a separate, pre-existing gRPC/BoringSSL issue unrelated to Gate 2 logic.
- The server's `aggregate_fit` received **1 result** (client-1) and **1 failure** (client-0 transport dropout). Gate 2 rejected the 1 result it did receive.
- `[Gate 2] No valid updates this round – skipping aggregation` confirms the defense mechanism: no tampered weights reached the FedAvg aggregator.
- **Conclusion:** Gate 2 correctly rejects updates signed with the wrong private key. An attacker who cannot access a client's legitimate private key cannot inject updates into the aggregation.

## Phase 4 – Gate 3: Behavioural Anomaly Detection (Label-Flipping Attack)

**Date:** 2026-02-28  
**Status:** ✅ PASSED – attacker detected and rejected in round 3

### Objective

Demonstrate that a malicious client who **passes** both Gate 1 (mTLS) and Gate 2 (RSA-PSS signature) can still be caught by **Gate 3** (delta-cosine behavioural anomaly detection).

The attack used is **Label Flipping**: the malicious client inverts all CIFAR-10 labels during local training (`label → 9 − label`, e.g. class 0 ↔ class 9). This causes the client's weight update to point in the **opposite** direction to honest training, which Gate 3 detects using the sign of the cosine similarity between the two clients' gradient deltas.

### Setup

| Item | Detail |
|------|--------|
| Framework | Flower `1.13.1` + PyTorch `2.5.1` + TorchVision `0.20.1` |
| Strategy | `ZeroTrustFedAvg` (Gate 2 + Gate 3) |
| Dataset | CIFAR-10 (IID partition, 25 000 samples/client) |
| Model | `CifarCNN` – 3 × Conv → 2 × FC → 10 classes |
| Rounds | 3 |
| Clients | 1 honest (`client-0`, `client.py`) + 1 malicious (`malicious`, `client_malicious.py`) |
| Environment | Docker Compose – staggered start (`sleep 5` honest, `sleep 12` malicious), NVIDIA CUDA 12.4 |

### Docker Services

| Container | Image | Command | Notes |
|-----------|-------|---------|-------|
| `fl-server` | `zt-pipeline-server` | `python server.py` | `NUM_CLIENTS=2`, `MIN_CLIENTS=2` |
| `fl-client-0` | `zt-pipeline-client-0` | `sleep 5 && python client.py` | `CLIENT_ID=0` – honest |
| `fl-malicious` | `zt-pipeline-malicious` | `sleep 12 && python client_malicious.py` | `CLIENT_ID=1`, `ATTACK_MODE=label_flip`, `LOCAL_EPOCHS=2` |

### Attack Flow Per Round

```
Malicious client:
  1. Receives global model from server  ← Gate 1 ✓ (valid mTLS cert)
  2. Trains 2 epochs with flipped labels  ← anti-correlated gradient
  3. Signs poisoned update with legitimate RSA key  ← Gate 2 ✓ (valid signature)
  4. Sends to server

Server Gate 3:
  - Round 1 (no Δ reference): Z-score fallback → both accepted
  - Round 2: cos_sim(Δ₀, Δ₁) = 0.0167 (barely positive) → both accepted
  - Round 3: cos_sim(Δ₁, Δ₀) = -0.0832 < 0 → client-1 REJECTED ✓
```

### Training Results

| Round | Gate 2 | Gate 3 Method | cos_sim | Accepted | Rejected | Eval Accuracy |
|-------|--------|--------------|---------|----------|----------|--------------|
| 1 | 2/2 ✓ | Z-score (no ref) | — | 2 | 0 | 20.51% |
| 2 | 2/2 ✓ | Delta cosine | **0.0167** | 2 | 0 | 22.63% |
| 3 | 2/2 ✓ | Delta cosine | **−0.0832** | 1 | **1 (client-1)** | **68.02%** |

**Total runtime:** 29.95 s  
**Aggregate failures:** 0 / 3 rounds (all clients completed training each round)

> **Round 3 accuracy jumps from 22.63 % → 68.02 %** because Gate 3 excluded the poisoned update.  
> Without Gate 3 (or in rounds 1–2 where the attacker was not yet detectable), the attack dragged accuracy far below the Phase 1 baseline (~70 %).

### Key Log Lines

```
# Gate 1 – malicious client presents its valid cert, passes authentication:
fl-malicious | [MALICIOUS 1] 🔒 mTLS certs loaded  → Gate 1 WILL pass

# Gate 2 – poisoned update carries a valid RSA-PSS signature, passes:
fl-server    | [Gate 2] ✓  Signature VALID for client-0
fl-server    | [Gate 2] ✓  Signature VALID for client-1
fl-server    | [Gate 2] Result: 2 passed, 0 rejected

# Gate 3 – round 3, anti-correlated deltas detected:
fl-server    | [Gate 3] Global reference available – using delta cosine similarity
fl-server    | [Gate 3] Delta norms: {'1': '10.524429', '0': '7.776110'}
fl-server    | [Gate 3] cos_sim(client-1 Δ, client-0 Δ) = -0.0832
fl-server    | [Gate 3] 🚨 SECURITY ALERT: anti-correlated deltas (cos_sim=-0.0832 < 0)
fl-server    | [Gate 3] 🚨 SECURITY ALERT: client-1 REJECTED (largest delta norm 10.524429)
fl-server    | [Gate 3] Result: 1 accepted, 1 rejected ['1']
```

### Observations

- **Gate 1 & Gate 2 are necessary but not sufficient.** The label-flip attacker held a valid CA-signed certificate and a legitimate RSA private key, so both cryptographic checks passed in every round.
- **Gate 3 provides the missing behavioural layer.** By tracking the global model after each aggregation and computing the cosine similarity of **per-round weight deltas**, the server detected that client-1's update pointed in the opposite direction to client-0's — a hallmark of label flipping.
- **Round 1 is a blind spot** — without a previous global snapshot, Gate 3 falls back to Z-score on raw weight norms.  With only 2 clients, the Z-score is always 1.0 for both (mathematical property of 2-point sets); detection is impossible in round 1 regardless of attack strength.  This is acceptable: one poisoned round from a random init has minimal impact.
- **Detection sharpens over rounds.** cos_sim trended from +0.017 (round 2) to −0.083 (round 3) as both clients converged on more distinct feature representations.  Increasing `LOCAL_EPOCHS` or `NUM_ROUNDS` would make the signal clearer earlier.
- **Rejection is correct.** The server picked client-1 (the attacker) for rejection because its delta norm was larger (10.52 vs 7.78) — the attacker's 2 local epochs produced a stronger deviation.
- **Post-rejection accuracy recovery.** After excluding the poisoned update in round 3, the FedAvg of one honest client yielded 68.02 % accuracy — consistent with the Phase 1 baseline (~70 %) and confirming the attacker was the source of degradation.
- **Known transient issue:** The BoringSSL `TSI_DATA_CORRUPTED` / `BAD_RECORD_MAC` error may appear if both clients send large weight tensors simultaneously over mTLS. Staggering client start times (5 s vs 12 s) eliminates the race condition.
